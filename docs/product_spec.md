# OfferCheck 产品完整功能规格

> 本文档面向 UI/UX 设计师。描述所有功能模块、交互逻辑和需要展示的内容，不涉及布局。
> 技术实现状态：Stage 1（选岗调研）和 Stage 4（Offer 证伪）引擎已完成，前端 UI 待设计实现。

---

## 一、产品定位与目标用户

**OfferCheck** 是面向海外/远程求职者的「全程怀疑型研究管家」。目标用户是**面向海外或远程岗位求职的用户**（留学生、跨境求职者、远程开发者），英文界面优先。

核心差异点：竞品（如求职方舟）做「公司实体工商信用查分」，OfferCheck 做「这次具体招聘机会本身」的主动证伪——覆盖冒名顶替（伪造 offer、仿冒域名邮箱、冒充 HR）这一最高频诈骗模式，这是纯靠企查查无法发现的。

**用户旅程**：从决定投哪个岗位 → 投递前优化简历 → 沟通过程持续监测 → 收到 offer 最终裁定，四个阶段由同一套自主调查引擎驱动，不是四个孤立功能。

---

## 二、整体功能地图

产品包含四个阶段入口 + 三套贯穿全程的共用系统：

**四个阶段入口**（按求职时间顺序）：
1. **选岗调研**（Stage 1）：投递前调研目标公司/岗位真实性
2. **简历定向**（Stage 2）：针对具体 JD 给出可执行的定向修改建议
3. **沟通证伪**（Stage 3）：核实招聘方身份、检测异常沟通模式
4. **Offer 证伪**（Stage 4）：对最终 offer/合同做深度交叉验证，输出靠谱/存疑/大概率有坑

**三套贯穿全程的共用系统**：
- **实时调查 Trace**：每一步调查操作的实时可见日志
- **结果卡片**：裁定/建议 + 证据链 + 待确认事项
- **来源引用系统**：所有结论必须绑定真实检索来源

## 三、阶段一：选岗调研

### 功能描述

用户给出候选公司或职位，Agent 自主联网调查，帮助判断值不值得投。核心价值是「投递前先做尽职调查」，避免把精力花在僵尸岗或问题公司上。

调查范围不是固定脚本，而是根据发现动态扩展：发现融资问题会深挖融资细节，发现域名异常会做 whois 核查，证据不足时自动换角度继续查。

### 输入模块

需要收集的用户信息：

- **目标公司名称**（必填）：如「Anthropic」、「字节跳动」
- **职位名称**（可选）：如「Senior Software Engineer」
- **简历关键信息或求职方向**（可选）：用于判断岗位与用户匹配度，可以是几句话描述，不需要上传完整简历
- **补充上下文**（可选）：如 JD 链接、HR 联系邮箱、岗位发布时间等

### Agent 自主调查的维度

Agent 会根据具体情况动态决定查哪些，但通常覆盖：
- 公司业务真实性（是否有真实产品/服务在运营，而非空壳）
- 融资健康度（最近融资时间、轮次、是否有倒闭/裁员迹象）
- 团队背景（创始人/核心管理层的可查公开信息）
- 岗位真实性（是否为长期挂着不招人的「僵尸岗」，岗位发布时间与公司现状是否匹配）
- 舆论风险（负面新闻、用户投诉、欠薪记录等）

### 输出内容（结果卡片需展示）

- **综合建议标签**：三选一——「值得投递」/ 「谨慎投递」/ 「建议放弃」
- **建议理由**：2-4 条核心依据，每条必须绑定具体来源（来源可点击）
- **公司真实性评估**：一句话结论 + 关键发现（如「成立 8 年，主营产品 X 在 App Store 可查，评分 4.2」）
- **岗位健康度**：是否为僵尸岗的判断 + 依据（如「该岗位已持续发布 14 个月，可能为常年虚开」）
- **融资/经营状况**：如有公开信息，显示最近融资时间/轮次；如无，显示「未找到公开融资记录」（不留空白）
- **风险提示**：如发现负面信息，逐条列出（如「2024 年 3 月有集体诉讼记录，涉及合同纠纷」）
- **待用户自行确认事项**：Agent 无法通过公开信息核实、但重要到需要用户主动去问的点（如「建议直接拨打公司官网电话确认 HR 联系方式是否一致」）
- **来源列表**：本次调查实际访问并引用的所有来源，可折叠展开

### 状态与边界

- 公司太小、信息完全不可查：显示「未找到足够公开信息」，列出已尝试的查询路径，建议用户补充 JD 链接后重试
- 发现严重风险：建议标签变为「建议放弃」，风险提示置顶高亮显示

## 四、阶段二：简历定向

### 功能描述

针对一个具体 JD，分析用户简历与该岗位的差距，给出可执行的定向修改建议。**不是全文重写简历**，是「这几点要突出/这几点要补」的精准清单，每条建议附依据。

### 输入模块

- **JD 文本**（必填）：粘贴岗位描述全文
- **用户简历文本**（必填）：粘贴简历全文（或关键部分）
- **目标说明**（可选）：用户希望重点强调的方向或顾虑

如果用户已在阶段一做过选岗调研，且调研的是同一家公司，已查到的公司信息可以自动带入，不需要重复输入公司背景。

### Agent 分析的内容

- 岗位关键词与简历现有内容的匹配/缺口分析
- 简历中哪些经历与 JD 最相关（应突出但目前未突出）
- 简历中哪些关键点在 JD 中重点提及但目前缺失（应补充）
- ATS（简历筛选系统）关键词覆盖度分析

### 输出内容（结果卡片需展示）

- **优先级排序的修改清单**：每条包含「要突出什么」或「要补充什么」+ 具体理由（关联 JD 中的哪句话）
- **关键词覆盖分析**：JD 中提到的核心技能关键词，哪些简历已有、哪些缺失
- **TOP 3 最高优先级改动**：突出显示最重要的 3 条建议
- 不输出改写后的完整简历

## 五、阶段三：沟通证伪

### 功能描述

用户粘贴招聘方发来的消息或面试邀约，Agent 核实招聘方身份、检测异常沟通模式（预付款要求、无面试直接录用、异常紧迫感等），必要时主动联网查证。

这是持续陪伴性质的功能：整个沟通过程中，用户可以多次粘贴新收到的消息，Agent 会结合上下文累积判断。

### 输入模块

- **招聘方消息/聊天记录**（必填）：粘贴文字记录，或描述沟通内容；如果是截图，可上传图片
- **招聘方信息**（可选）：HR 姓名、邮箱、电话、微信号、所在平台（LinkedIn/Boss/邮件等）
- **背景上下文**（可选）：是否已面试、面试形式（视频/电话/无面试）、是否有 offer 意向
- **上一阶段的调查结果**：如果用户在阶段一调研过这家公司，已有发现自动带入，不需重填

### Agent 核查的维度

- 招聘方邮箱域名是否与公司官网匹配（如 HR 自称字节跳动但邮箱是 bytedance-hr.com）
- 沟通模式异常检测：
  - 是否要求缴纳押金/设备费/培训费
  - 是否无面试直接发 offer
  - 是否制造异常紧迫感（「今天必须确认」「名额只剩一个」）
  - 是否要求使用加密货币、礼品卡、转账等非常规支付方式
  - 是否索要过多个人信息（身份证、银行卡）在不应该的阶段
- 如发现异常，主动联网核查该公司/HR 的公开信息

### 输出内容（结果卡片需展示）

- **风险等级标签**：「未发现风险」/ 「存在可疑信号」/ 「高风险，强烈建议谨慎」
- **检测到的红旗列表**：逐条列出，每条说明「异常是什么」+ 「为什么这是风险信号」+ 来源（如「招聘方邮箱域名 bytedance-hr.com 注册于 2024 年 3 月，与字节跳动官方域名 bytedance.com 不同」）
- **未发现风险时**：明确显示「已核查 X 项，未发现明显异常信号」（不留空白）
- **建议下一步动作**：具体可执行的建议（如「建议拨打字节跳动官方招聘热线 xxx-xxxx 核实该 HR 身份」）
- **待确认事项**：需用户自行判断的模糊点

## 六、阶段四：Offer 证伪（核心功能）

### 功能描述

用户收到正式 offer 或合同后，进行最深度的交叉验证，专门寻找欺诈、冒名顶替、条款陷阱。这是整个产品的核心能力，其他三个阶段都是铺垫。

输出的「靠谱/存疑/大概率有坑」三态裁定是这个功能的标志性结果。三态都是合法结论：
- **靠谱**：调查到的证据支持 offer 真实可信（不代表工作完全没问题，只是从真实性角度通过调查）
- **存疑**：发现矛盾或无法核实的关键点，建议用户进一步确认后再决定
- **大概率有坑**：发现欺诈/冒名/严重矛盾的明确证据（这是最严重但完全合法的裁定，有证据支持就该输出）

### 输入模块

- **Offer 或合同文本**（必填）：粘贴全文，尽量包含公司名称、岗位、薪资、入职地点、合同主体
- **HR 联系信息**（可选）：邮箱、电话、微信等，用于核实身份
- **公司网站或职位链接**（可选）
- **全程沟通背景**（可选）：已经历的面试情况、之前沟通中的异常（可以直接粘贴之前的沟通内容）
- 如果用户前三个阶段已有调查结果，自动带入，Agent 在此基础上深化，不重复查已查过的内容

### Agent 调查的维度

- **公司真实性核查**：注册信息、营业执照号、实际运营状态与 offer 描述的一致性
- **域名核查**：offer 邮箱域名、公司官网的注册时间/注册者/历史；识别「字节跳动」和「bytedance-hr.com」这类山寨域名陷阱
- **合同主体核查**：签约主体是否与 offer 描述的公司一致，是否存在中间层/壳公司
- **薪资条款核查**：薪资结构是否异常（超高薪酬+要求先缴费是典型诈骗模式）
- **冒名顶替检测**：是否是真实公司的冒名版本（真实公司信用很好，但有人拿它名义行骗）
- **异常条款检测**：入职前付款要求、押金、设备费、培训费等
- **Offer 核实建议**：哪些可以通过公开渠道（LinkedIn、企业官网、电话）直接核实

### 输出内容（结果卡片需展示）

**核心裁定区域**（最显眼的展示区）：
- 裁定结论：「靠谱」/ 「存疑」/ 「大概率有坑」，三态视觉上需明显区分
- 裁定理由：1-2 句核心说明，说明裁定依据的最关键发现

**红旗清单**（存疑/有坑时显示）：
- 逐条列出发现的风险点
- 每条包含：问题描述 + 证据（附来源链接，可点击）
- 如无红旗：显示「本次调查未发现明显风险信号」

**正向证据列表**（靠谱时显示）：
- 支持 offer 真实性的发现（如「公司官网域名注册于 2018 年，与官方一致」）
- 每条附来源

**待用户自行确认事项**（三种裁定都可能有）：
- Agent 无法通过公开渠道核实、需要用户主动行动的点
- 具体操作建议（「拨打官方客服电话 xxx 核实 HR 身份」而非模糊的「建议核实」）

**来源列表**（可折叠展开）：
- 本次调查实际访问并使用的所有来源，带链接
- 显示每条来源支持了哪个判断点

## 七、贯穿全程：实时调查 Trace

这是整个产品的透明度核心，让用户看到 Agent 每一步在做什么，建立信任感，同时帮用户判断调查是否充分。

### 触发时机

用户点击「开始调查」后，Trace 立即开始实时更新，**不等全部完成后才显示**。每一步完成后，Trace 区域新增一条记录，用户无需等待即可看到调查在推进。

### Trace 的每一条记录包含

**基础信息**（始终显示）：
- 步骤序号（第 N 步）
- 行动描述：自然语言描述这一步在做什么，而不是技术参数（如「搜索字节跳动官方招聘信息」而非「web_search(bytedance official recruitment 2024)」）
- 工具类型标识：搜索 / 网页抓取 / 域名查询 / 图片识别 等，视觉上可区分
- 执行状态：成功 / 未找到有效内容 / 缓存命中（已在本次调查中查询过，直接复用结果）

**展开后的详细内容**（默认折叠，点击展开）：
- 具体查询内容（如搜索关键词、访问的 URL）
- 返回结果的摘要（不是全文，是关键信息提取）
- 对于搜索结果：展示前 3-5 条结果的标题和链接
- 对于网页抓取：展示提取到的关键段落
- 对于域名查询：展示注册时间、注册者、注册商等

### Trace 中工具类型的说明

产品当前拥有以下调查工具，每种在 Trace 中需要有对应的视觉标识：

| 工具 | 用途 | Trace 中展示 |
|---|---|---|
| 网络搜索 | 通用信息搜索 | 搜索关键词 + 结果摘要 |
| 网页抓取 | 读取特定网页内容 | 访问 URL + 提取内容摘要 |
| 域名查询 | 查 whois 注册信息 | 域名 + 注册时间/注册者 |
| 图片识别 | 读取截图/照片中的文字 | 图片描述 + 提取文本 |
| PDF 解析 | 读取合同/offer PDF | 文件名 + 提取关键段落 |
| 维基百科 | 查公司/人物背景 | 查询词 + 摘要 |

### Trace 的运行状态

- **调查进行中**：最新一步显示「正在执行...」动效；显示「第 N 步 / 最多 M 步」进度
- **调查完成**：全部步骤定格显示，结果卡片出现
- **步数用尽未完成**：显示已查到的内容 + 说明「由于信息复杂度，已达到调查步数上限，以下为基于现有证据的综合判断」；结果卡片仍然出现，但说明调查未能完全穷尽
- **调查中断/错误**：显示到哪一步失败，原因说明，提供重试按钮

### Trace 的交互

- 默认状态：每条 Trace 只显示基础信息（步骤序号 + 行动描述 + 状态）
- 点击任意一条 Trace：展开查看该步的详细内容
- 用户可同时展开多条 Trace
- Trace 区域默认跟随滚动到最新步骤；用户手动向上滚动后暂停跟随，直到回到底部

### 缓存命中的展示

当 Agent 检测到某个查询在本次调查中已执行过时（防止重复浪费），该步 Trace 显示「缓存命中」状态 + 说明「此查询已在第 X 步执行过，直接使用已有结果」。这让用户知道 Agent 不是在重复做无用功。

## 八、贯穿全程：来源引用系统

所有阶段的结论必须绑定来源，这是产品的核心信任机制。

### 来源的显示规则

- 结果卡片中每条判断后面必须有来源标注（可点击跳转原始页面）
- 来源必须是本次调查中 Agent 真实访问并提取到内容的 URL，不允许展示未实际访问的来源
- 来源链接如果在中国大陆因 GFW 无法直接访问（如 LinkedIn），在来源旁标注「在中国大陆需 VPN 访问」，不因此删除来源

### 无法核实的来源处理

如果 Agent 生成的某条结论引用了在调查中未实际访问到的来源，该来源标注「⚠️ 未经验证」，并在结果卡片中有一条整体提示「部分引用来源未能在本次调查中核实，请用户自行判断」。

### 来源列表（可独立展开的区块）

每次调查结束后，在结果卡片下方提供一个完整来源列表：
- 本次调查访问的所有有效来源
- 每条显示：域名 + 页面标题 + 该来源支持了哪个判断点
- 可折叠，默认折叠

## 九、贯穿全程：跨阶段记忆

### 记忆的作用

同一用户在同一次求职过程中，四个阶段的调查结果应该互相传承：

- 阶段一查明的公司信息（成立时间、融资状况、真实业务）→ 阶段三/四不需要重查
- 阶段三发现的 HR 邮箱异常 → 阶段四做 offer 证伪时自动带入已有发现
- 阶段四的裁定 → 沉淀为该公司的历史调查记录，下次调查同一公司时可参考

### 记忆在 UI 上的体现

- 在阶段三/四的输入界面：如果检测到用户已调研过该公司（通过公司名匹配），显示提示「已找到您对 [公司名] 的调查记录（来自阶段一，X 天前），已自动带入」，用户可以选择是否使用
- 在 Trace 中：如果某步直接使用了历史记忆而非重新查询，显示「引用历史记录（来自 [日期]）」而非假装是新查询
- 跨阶段记忆不跨越不同公司/岗位

## 十、图片/截图输入能力

阶段三（沟通证伪）和阶段四（Offer 证伪）支持用户上传图片，适用场景：
- 微信/WhatsApp 等聊天软件的截图（文字消息、offer 图片）
- Offer letter 的 PDF 或图片版本
- 招聘方发来的带公司 logo 的图片

图片上传后，Agent 会先用图像识别提取文字，然后对提取的文字内容进行分析。在 Trace 中会有一步「图片文字识别」记录，展开后显示提取到的文本内容，用户可以确认提取是否准确。

## 十一、空状态与边界情况说明

设计需要覆盖以下边界情况（均不应留白）：

| 情况 | 应展示的内容 |
|---|---|
| 公司信息完全查不到 | 「未找到该公司的公开信息，可能是成立时间短的小公司、内部孵化项目或信息确实有限」+ 已尝试的查询路径 + 建议用户补充 JD 链接后重试 |
| Agent 在步数内未查完 | 「本次调查已达到步数上限，以下为基于现有 X 项证据的综合判断，可能不完整」|
| Red Flags 列表为空（靠谱时）| 「本次调查未发现明显风险信号（已核查 X 项）」，不允许留空 |
| 调查被中断/API 错误 | 显示已完成的步骤 + 明确的错误提示 + 「重新调查」按钮 |
| 某来源因网络原因无法访问 | 该步 Trace 标注「无法访问，已跳过」，Agent 会换用其他来源继续调查 |
| 「存疑」裁定 | 不应显示为「失败」或「错误」，存疑是正当的裁定结果，视觉上不应与错误状态混淆 |

## 十二、语言与本地化

- **界面语言**：英文优先（目标用户是海外/远程求职者）
- **结果内容语言**：根据调查目标自动适应。调查中国公司的结果可能是中文信息，调查美国公司的结果可能是英文，不强制统一
- **Trace 中的行动描述**：始终用英文展示给用户（即使底层查询是中文关键词）
- **裁定标签**：英文展示（「Looks Legit」/ 「Suspicious」/ 「Likely a Scam」），对应中文裁定的「靠谱/存疑/大概率有坑」

## 十四、竞品参考（设计灵感来源）

以下产品有值得参考的交互模式：

- **Perplexity Deep Research**：实时步骤流展示 + 来源列表侧栏，调查中即可看到进度
- **OpenAI Deep Research**：可折叠的步骤详情展示方式，完成后的报告形式
- **Claude Research / Claude Artifacts**：证据链接内联在文本中，点击可直达来源
- **求职方舟（对比参考）**：静态报告形式，代表「不要做成这样」——没有实时过程，没有来源验证，没有动态调查

差异化核心：OfferCheck 的 Trace 不是装饰，是产品功能的一部分——它让用户能判断调查是否充分，也是「主动证伪」和「静态查分」最明显的产品区分。
